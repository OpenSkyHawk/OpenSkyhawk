# A4ECGenerator — Technical Specification

**Status:** Done
**FirmwarePlan ref:** `FirmwarePlan/04-dcs-bios-integration.md#generated-headers`,
`FirmwarePlan/10-implementation-plan.md#phase-1--core-shared-contracts-prerequisite-for-bridgegateway`
**Depends on:** *(build tool — no firmware library dependency)*

---

## Responsibility

Python 3 script that parses DCS-BIOS A-4E-C control definitions from a committed JSON
snapshot and emits three C++ headers consumed by the OpenSkyhawk firmware plus one generated
gap report:

| Generated file | Included by | Contents |
|----------------|-------------|----------|
| `A4EC_CmdIds.h` | PanelGroup sketches, PanelBridge | `#define DCSIN_*` compact transport IDs for inputs |
| `A4EC_OutputIds.h` | PanelGroup sketches with LED/output objects | `#define A_4E_C_*` / `A_4E_C_*_AM` — DCS-BIOS published identifiers for output addresses and masks |
| `A4EC_InputMap.h` | PanelBridge only | `DcsBiosInputEntry[]` dispatch table (struct + type constants + sorted array) |
| `GENERATOR_GAPS.md` | Humans only | Every skipped control with the reason it was not generated |

All generated files are **committed to the repo** alongside the generator. The PlatformIO
firmware build depends only on the committed header files — no Python or JSON processing at
compile time.

Does **not** run automatically during the PlatformIO build. Regenerate manually when the
DCS-BIOS A-4E-C JSON changes, then commit the updated headers and reflash all boards.

Does **not** own the `DcsBiosInputEntry` struct definition — that struct is emitted verbatim
into `A4EC_InputMap.h` so the generated file is self-contained with no external `#include`.
PanelBridge.h includes `A4EC_InputMap.h`, not the other way around.

## Running the Generator

From the repo root:

```bash
python tools/gen_a4ec/gen_a4ec.py
```

The script reads `tools/gen_a4ec/data/A-4E-C.json` and writes directly to
`Firmware/Libraries/A4EC/`. Existing headers are overwritten. `GENERATOR_GAPS.md` is always
regenerated — do not edit it manually.

After running, commit all changed files in `Firmware/Libraries/A4EC/` and reflash all boards.
PanelBridge and every PanelGroup node must run firmware built from the same header revision —
`DCSIN_*` IDs are not backward-compatible across regenerations.

> **Warning:** regenerating after a JSON update reassigns `DCSIN_*` IDs for any controls
> whose names were added, removed, or renamed. `A_4E_C_*` output identifiers mirror the
> DCS-BIOS published names and are stable across JSON updates unless DCS-BIOS itself
> reassigns them (rare). All boards must be reflashed together after any regeneration.

## Updating the JSON Snapshot

Copy the JSON from the DCS-BIOS A-4E-C mod installation, then regenerate:

```bash
cp "$DCS_PATH/Mods/aircraft/A-4E-C/DCS-BIOS/A-4E-C.json" tools/gen_a4ec/data/
python tools/gen_a4ec/gen_a4ec.py
```

Then commit both the updated `A-4E-C.json` and all changed files in `Firmware/Libraries/A4EC/`.

## PlatformIO Integration

`A4EC` is a header-only library consumed via `lib_deps`. No `extra_scripts` entry is needed.

**PanelGroup `platformio.ini`:**
```ini
lib_deps =
    file://../../Libraries/A4EC
    file://../../Libraries/CANProtocol
    file://../../Libraries/STM32Board
    file://../../Libraries/PanelGroup
```

**PanelBridge `platformio.ini`:**
```ini
lib_deps =
    file://../../Libraries/A4EC
    file://../../Libraries/CANProtocol
    file://../../Libraries/STM32Board
```

PanelGroup sketches `#include <A4EC_CmdIds.h>` for input declarations. Sketches that also
declare LED or other output objects `#include <A4EC_OutputIds.h>` for `A_4E_C_*` constants.
PanelBridge `#include`s `A4EC_CmdIds.h` and `A4EC_InputMap.h`; it does not need
`A4EC_OutputIds.h` (it forwards native DCS-BIOS addresses directly as CTRL_BCAST controlIds).

## Generated Output Specification

### A4EC_CmdIds.h

No external `#include` required — constants only.

```cpp
// A4EC_CmdIds.h — AUTO-GENERATED by tools/gen_a4ec/gen_a4ec.py
// DO NOT EDIT — regenerate using: python tools/gen_a4ec/gen_a4ec.py
// DCS-BIOS source: tools/gen_a4ec/data/A-4E-C.json
// Generated: <ISO-8601 timestamp>
// Control count: <N>

#pragma once

// DCSIN_* — compact CAN transport IDs for DCS-BIOS A-4E-C input controls.
// Range: 0x8001 – 0x86FF (see FirmwarePlan/04-dcs-bios-integration.md).
// IDs are assigned in alphabetical order of the DCS-BIOS control name.

#define DCSIN_ACLS_ENGAGE          0x8001
#define DCSIN_ANTI_SKID_SWITCH     0x8002
// ... one #define per supported A-4E-C input control, alphabetical order
```

Control names are taken verbatim from the JSON top-level key (the DCS-BIOS control name).
The `DCSIN_` prefix is prepended and the name is converted to uppercase.

---

### A4EC_OutputIds.h

No external `#include` required — constants only. Included by PanelGroup sketches that
declare LED or other output objects. Not needed by PanelBridge.

Constant names are taken **verbatim** from the JSON `address_mask_shift_identifier` field
(e.g. `A_4E_C_AOA_GREEN`). The mask constant uses the `address_mask_identifier` field when
present (e.g. `A_4E_C_AOA_GREEN_AM`); if absent, `_AM` is appended to the address constant
name. These identifiers are the ones published by DCS-BIOS and displayed by debug tools such
as Bort — using them verbatim means a developer can search the codebase with the exact string
they see in a debug tool.

A `_SHIFT` constant is **not** generated at this stage. Shift is only needed by integer
output classes (gauges, counters) where the raw value must be extracted from a packed word.
Those classes are not yet specced; `_SHIFT` will be added to the generator when they are.

```cpp
// A4EC_OutputIds.h — AUTO-GENERATED by tools/gen_a4ec/gen_a4ec.py
// DO NOT EDIT — regenerate using: python tools/gen_a4ec/gen_a4ec.py
// DCS-BIOS source: tools/gen_a4ec/data/A-4E-C.json
// Generated: <ISO-8601 timestamp>
// Output count: <N>

#pragma once

// Output address and mask constants for A-4E-C cockpit indicators and gauges.
// Constant names match the DCS-BIOS published identifiers exactly — the same names
// shown in Bort and other DCS-BIOS debug tools.
//
// For each output, two constants are defined:
//   <address_mask_shift_identifier>     — uint16_t DCS-BIOS output address
//   <address_mask_identifier> (_AM)     — uint16_t bitmask for the relevant bits
//
// Usage with LED:
//   OpenSkyhawk::LED aoaGreen(A_4E_C_AOA_GREEN, A_4E_C_AOA_GREEN_AM, pin);

#define A_4E_C_AOA_GREEN     0x8470  // AOA Indexer (green)
#define A_4E_C_AOA_GREEN_AM  0x0001

#define A_4E_C_AOA_RED       0x8470  // AOA Indexer (red) — same address word, different bits
#define A_4E_C_AOA_RED_AM    0x0004

#define A_4E_C_ARM_MASTER    0x84D0  // Master Arm
#define A_4E_C_ARM_MASTER_AM 0x4000

// ... one pair per supported A-4E-C output control, alphabetical order by identifier
```

**Decimal-to-hex conversion:** all numeric values in the JSON (`address`, `mask`) are stored
as plain decimal integers (e.g. `address: 34048`, `mask: 16384`). The generator must convert
these to hex when emitting `#define`s (e.g. `0x84D0`, `0x4000`) — hex is far more readable
for addresses and bitmasks and matches the convention in the rest of the firmware.

**Why native addresses instead of compact IDs?** `DCSIN_*` compact IDs are necessary for
inputs because DCS-BIOS control names (strings) are too large to send over CAN. For outputs,
PanelBridge already has the native DCS-BIOS address from the protocol — forwarding it directly
as `controlId` in CTRL_BCAST requires no translation table. Using the published `A_4E_C_*`
identifiers ensures the sketch value matches what PanelBridge sends, and aligns with what
DCS-BIOS debug tools (e.g. Bort) display.

### A4EC_InputMap.h

Self-contained — defines its own struct and type constants. PanelBridge includes this header
and uses the exported symbols directly; no other header is required for the dispatch table.

```cpp
// A4EC_InputMap.h — AUTO-GENERATED by tools/gen_a4ec/gen_a4ec.py
// DO NOT EDIT — regenerate using: python tools/gen_a4ec/gen_a4ec.py
// Include from PanelBridge only — PanelGroup sketches do not need this file.
// Generated: <ISO-8601 timestamp>

#pragma once
#include <A4EC_CmdIds.h>

// ── Input dispatch type codes ─────────────────────────────────────────────────

namespace InputType {
    static constexpr uint8_t SWITCH        = 0;  ///< Switch2Pos — sends 0 or 1
    static constexpr uint8_t ACTION        = 1;  ///< ActionButton — sends 1 (press only)
    static constexpr uint8_t ENCODER       = 2;  ///< RotaryEncoder — sends 0=CCW, 1=CW
    static constexpr uint8_t ACCEL_ENCODER = 3;  ///< RotaryAcceleratedEncoder — 0–3
    static constexpr uint8_t MULTIPOS      = 4;  ///< Switch3Pos/SwitchMultiPos/AnalogMultiPos/RotarySwitch/AnalogInput
}

// ── Dispatch entry ────────────────────────────────────────────────────────────

struct DcsBiosInputEntry {
    uint16_t    cmdId;      ///< DCSIN_* compact ID — matches A4EC_CmdIds.h constant
    uint8_t     type;       ///< One of InputType::* — determines PanelBridge dispatch
    const char* name;       ///< DCS-BIOS control name for sendDcsBiosMessage()
    const char* arg0;       ///< value=0 argument string ("0", "DEC", position string)
    const char* arg1;       ///< value=1 argument string ("1", "INC"); nullptr for ACTION/MULTIPOS
    const char* arg0fast;   ///< ACCEL_ENCODER: fast decrement arg; nullptr for all other types
    const char* arg1fast;   ///< ACCEL_ENCODER: fast increment arg; nullptr for all other types
};

// ── Dispatch table — sorted ascending by cmdId ────────────────────────────────
// Binary search by PanelBridge. Flash cost ~6–8 KB. RAM cost: zero.

static const DcsBiosInputEntry A4EC_INPUT_MAP[] = {
    { DCSIN_ACLS_ENGAGE,      InputType::ACTION,  "ACLS_ENGAGE",      "1",   nullptr, nullptr, nullptr },
    { DCSIN_ANTI_SKID_SWITCH, InputType::SWITCH,  "ANTI_SKID_SWITCH", "0",   "1",     nullptr, nullptr },
    // ... one entry per supported A-4E-C input control, sorted by cmdId
};

static constexpr uint16_t A4EC_INPUT_MAP_SIZE =
    sizeof(A4EC_INPUT_MAP) / sizeof(A4EC_INPUT_MAP[0]);
```

The table is sorted by `cmdId`. Because IDs are assigned alphabetically, alphabetical name
order equals ascending `cmdId` order — the sort is effectively free.

## Testing

Two levels of testing: the generator script and the generated headers.

### Generator tests (Python)

```
tools/gen_a4ec/
├── tests/
│   ├── fixture.json        ← small hand-crafted JSON with one control of each mapping type
│   └── test_gen.py         ← runs generator against fixture; checks output content
```

`test_gen.py` runs the generator against `fixture.json` (not the real A-4E-C.json) and verifies:

| Test | What it checks |
|------|---------------|
| ID assignment | IDs start at 0x8001, increment by 1, in alphabetical order |
| SWITCH mapping | `set_state` + `max_value == 1` → `InputType::SWITCH`, arg0=`"0"`, arg1=`"1"` |
| ACTION mapping | `action` interface → `InputType::ACTION`, arg0 from JSON `argument`, arg1/arg0fast/arg1fast are `nullptr` |
| ENCODER mapping | single `fixed_step` → `InputType::ENCODER`, arg0=`"DEC"`, arg1=`"INC"` |
| ACCEL_ENCODER mapping | two `fixed_step` variants (slow + fast) → `InputType::ACCEL_ENCODER`, all four arg fields populated |
| MULTIPOS mapping | `set_state` + `max_value > 1` → `InputType::MULTIPOS` |
| Gap file emitted | `GENERATOR_GAPS.md` is written on every run, even when there are no skipped controls |
| Gap logging | unsupported mapping → not in A4EC_CmdIds.h; recorded in GENERATOR_GAPS.md with control name, interface/pattern, and reason |
| Paired boolean gap | guarded cover+switch pattern is skipped and logged as requiring `SwitchWithCover2Pos` / two-name mapping support |
| Sort order | A4EC_INPUT_MAP[] entries are sorted ascending by cmdId |
| Count constant | `A4EC_INPUT_MAP_SIZE` equals actual array length |
| Output address | emitted constant name matches JSON `address_mask_shift_identifier`; value matches `outputs[].address` (hex) |
| Output mask | emitted `_AM` constant name matches JSON `address_mask_identifier`; value matches `outputs[].mask` (hex) |
| Output multi-entry | control with multiple output entries emits one pair per entry using the JSON identifier |
| Output alphabetical | `A4EC_OutputIds.h` entries appear in alphabetical order by identifier |

Run from repo root:
```bash
python -m pytest tools/gen_a4ec/tests/test_gen.py
```

No external dependencies beyond Python 3 stdlib and `pytest`.

## Generator Algorithm

```
── Input pass ────────────────────────────────────────────────────────────────────
1. Load JSON from tools/gen_a4ec/data/A-4E-C.json
2. Collect all top-level keys (control names) that have an "inputs" array
3. Sort alphabetically by control name
4. Assign sequential IDs: 0x8001, 0x8002, …  (fail with error if count exceeds 0x86FF - 0x8001)
5. For each control:
   a. Inspect the "inputs" array (DCS-BIOS `interface` fields)
   b. Apply generator mapping rules (see table below) to select SWITCH/ACTION/ENCODER/ACCEL_ENCODER/MULTIPOS
   c. Extract arg0, arg1, arg0fast, arg1fast from the JSON values
   d. If the control cannot be mapped (unsupported interface combination): skip it,
      record name + reason in GENERATOR_GAPS list
6. Emit A4EC_CmdIds.h  — only the controls that were successfully mapped
7. Emit A4EC_InputMap.h — same set, sorted by cmdId (= alphabetical by name)

── Output pass ───────────────────────────────────────────────────────────────────
8. Collect all top-level keys (control names) that have an "outputs" array
9. Sort alphabetically by control name
10. For each control:
    a. Collect all entries in the "outputs" array; each entry has address, mask, shift_by
    b. If the control has exactly one output entry: emit one pair using
       address_mask_shift_identifier as the address constant name and
       address_mask_identifier as the mask constant name (_AM suffix)
    c. If the control has multiple output entries: emit one pair per entry, each
       using its own address_mask_shift_identifier and address_mask_identifier
    d. Controls with outputs that cannot be represented (no address field, etc.) are
       skipped and recorded in GENERATOR_GAPS list
11. Emit A4EC_OutputIds.h — all successfully mapped output controls, alphabetical order

── Shared ────────────────────────────────────────────────────────────────────────
12. Emit GENERATOR_GAPS.md — one entry per skipped input or output control with reason
```

### Generator Mapping Rules

| JSON `inputs[].interface` | Generated type | arg fields |
|------------------------------------|----------------|------------|
| `action` | `ACTION` | `arg0` = JSON `argument` value; `arg1/arg0fast/arg1fast` = nullptr |
| `fixed_step` (single entry) | `ENCODER` | `arg0` = `"DEC"`, `arg1` = `"INC"` |
| `set_state`, `max_value == 1` | `SWITCH` | `arg0` = `"0"`, `arg1` = `"1"` |
| `set_state`, `max_value > 1` | `MULTIPOS` | `arg0` unused (nullptr); value sent as `itoa(value)` |
| Two `fixed_step` entries (slow + fast) | `ACCEL_ENCODER` | `arg0/arg1` = slow dec/inc; `arg0fast/arg1fast` = fast dec/inc |
| Paired boolean controls (cover + switch) | **Skipped** | Logged in GENERATOR_GAPS.md |
| Any other unrecognised combination | **Skipped** | Logged in GENERATOR_GAPS.md |

These rules exactly mirror `FirmwarePlan/04-dcs-bios-integration.md#generator-mapping-rules`.

## GENERATOR_GAPS.md

Auto-generated file documenting every control the generator could not map. This file is
emitted on every generator run, even when empty, so reviewers can see that unsupported
controls were considered deliberately. Format:

```markdown
# A4EC Generator Gaps

Auto-generated by tools/gen_a4ec/gen_a4ec.py — DO NOT EDIT.
Last run: <ISO-8601 timestamp>
Source: tools/gen_a4ec/data/A-4E-C.json

## Unsupported Controls

| Control name | interface(s) / pattern | Reason |
|---|---|---|
| EXT_STORES_JETTISON | paired_boolean | Cover + switch share one DCS-BIOS address — SwitchWithCover2Pos not yet supported |
```

Commit `GENERATOR_GAPS.md` alongside the generated headers after each run. Unsupported
controls must be represented here; silent skips are generator bugs.

## Consumers of the Generated A4EC Library

The `Firmware/Libraries/A4EC/` output is a dependency of:

| Consumer | Headers used | Notes |
|---|---|---|
| **PanelGroup sketches** (e.g. `Center_Armament/main.cpp`) | `A4EC_CmdIds.h` | `DCSIN_*` constants used in input object declarations. |
| **PanelGroup sketches with outputs** (LEDs, gauges) | `A4EC_CmdIds.h` + `A4EC_OutputIds.h` | `A_4E_C_*` / `A_4E_C_*_AM` constants used in LED, IntegerOutput, and other output object declarations. Names match DCS-BIOS published identifiers. |
| **PanelBridge** | `A4EC_CmdIds.h` + `A4EC_InputMap.h` | Full dispatch table required for EVT routing. Does not need `A4EC_OutputIds.h` — forwards native DCS-BIOS addresses directly. See `PanelBridge.md`. |

The PanelGroup *library itself* does not depend on A4EC — only the per-panel sketches do.
`SimGateway` and the `PanelGroup` library do not depend on A4EC.
