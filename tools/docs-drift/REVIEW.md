# Weekly Docs Drift Review — Playbook

This is the procedure the **`weekly-docs-drift-review`** scheduled task follows. It is run
by a fresh Claude with no memory of prior runs, so everything needed is here. The job is to
catch *prose* drift that structured checks can't: new hardware that has no page, and design
decisions that changed in a source file but not on the published page.

Structured tables (NODE_IDs, etc.) are already kept in sync by transclusion +
`tools/check_node_ids.py` — **do not** re-check those here.

## Ground rules

- **Read-only on the repo.** This run opens or updates one GitHub issue. It does **not**
  edit docs, commit, or push.
- **Assume cwd is the repo root.** The scheduled-task prompt handles `cd` + `git fetch` +
  checkout of `origin/main`.
- **Ignore `docs/api/**`** — those pages are generated by mkdoxy; they are never drift.
- Output is a single issue labelled `docs-drift`. Re-running updates the same open issue
  rather than spawning duplicates.

## Source of truth → published docs

| Source of truth (what changed) | Published surface (what to keep in sync) |
|---|---|
| `docs/_source/**` (architecture, `hardware-standards.md`, `pcb-design-rules.md`, `kicad.md`, `controllers/`) | `docs/**` prose pages (excluding `docs/api/`, `docs/_source/`) |
| `CLAUDE.md` (project rules, decisions) | architecture / contributing / hardware pages |
| `Firmware/Panels/**`, `Firmware/Libraries/**`, `Firmware/NODE_IDS.md` | `docs/firmware/**`, panel pages |
| `PCB/**`, `CAD/**` (new boards / panels) | `docs/hardware/**`, `docs/panels/**`, kit pages |
| **GitHub Projects #2** (Controller Build, org `OpenSkyHawk`) — panel/controller past the schematic stage (B2+) | a matching page must exist under `docs/` |

## Step 1 — Establish the window

```bash
SINCE="7 days ago"
git log --since="$SINCE" --pretty='%h %ad %s' --date=short
```

If there are **no** commits in the window, post the "no drift" note (Step 5) and stop.

## Step 2 — Gather source-of-truth changes

```bash
SINCE="7 days ago"
git log --since="$SINCE" --name-only --pretty='%n=== %h %s' -- \
  docs/_source CLAUDE.md \
  Firmware/Libraries Firmware/Panels Firmware/NODE_IDS.md \
  PCB CAD
```

Read the actual diffs for anything that looks like a decision or a new component:

```bash
git log --since="$SINCE" -p -- docs/_source/hardware-standards.md CLAUDE.md \
  docs/_source/architecture.md docs/_source/pcb-design-rules.md
```

Also list directories that are **new** in the window (candidate new hardware with no page):

```bash
git log --since="$SINCE" --diff-filter=A --name-only --pretty=format: -- \
  PCB CAD Firmware/Panels | sort -u
```

## Step 3 — Compare against the published docs

For each changed source item, decide whether a published page is now wrong or missing.

- **New hardware → missing page.** A new `PCB/<Console>/<Group>/<Board>/`, new
  `CAD/<...>`, or new `Firmware/Panels/<Panel>/` should have a corresponding page under
  `docs/hardware/`, `docs/panels/`, or a kit page. Grep the docs for the board/panel name:
  ```bash
  grep -ril "<Board or Panel name>" docs --include='*.md' | grep -v '^docs/api/'
  ```
  No hit → flag as **missing page**.
- **Changed decision → stale page.** When a value or decision changed in a source file
  (e.g. a part number in `hardware-standards.md`, an MCU-variant rule, a design decision
  D1–D8 such as the CAD tool choice), grep the published pages for the **old** value and
  confirm none still assert it:
  ```bash
  grep -rin "<old value / old decision>" docs --include='*.md' | grep -v '^docs/api/'
  ```
  Any hit on a published page → flag as **stale decision**.
- **Contradiction.** If a published page states something that now conflicts with
  `docs/_source/` material, flag it with both locations.

Use judgement — only flag real, actionable mismatches. A reworded sentence in a source
file with no factual change is **not** drift.

## Step 4 — Build-page cross-check (best effort)

Panel and controller **build tracking lives in GitHub Projects** (org `OpenSkyHawk`), **not
Notion** — the legacy Notion Panels DB is research-reference only, so do **not** use it as
the build tracker. If GitHub Projects is reachable, list **Project #2 (Controller Build)**
items and confirm each controller/panel past the schematic stage (B2+) has a page under
`docs/`:

```bash
gh project item-list 2 --owner OpenSkyHawk --format json
```

Flag any built/in-progress controller or panel with no docs page. If the check can't run,
note "build-page cross-check skipped" in the issue and continue — do not fail. (Notion still
holds non-panel data — Tasks DB, project notes — but that is out of scope for this review.)

## Step 5 — Write the issue

Ensure the label exists (no-op if it already does):

```bash
gh label create docs-drift --color FBCA04 \
  --description "Docs site has drifted from a source of truth" 2>/dev/null || true
```

Find an existing open issue and reuse it:

```bash
gh issue list --label docs-drift --state open --json number,title
```

- **Findings exist:** build a markdown body (template below). If an open `docs-drift` issue
  exists, `gh issue edit <n> --body-file <file>` and add a dated comment noting the refresh;
  otherwise `gh issue create --label docs-drift --title "Weekly docs drift review — $(date +%F)" --body-file <file>`.
- **No findings:** if an open issue exists, comment "No new drift detected as of `<date>`.";
  if none exists, do nothing (don't open an empty issue).

### Issue body template

```markdown
# Docs drift — week ending <YYYY-MM-DD>

Reviewed source-of-truth changes from the last 7 days against the published docs.
Ignored `docs/api/` (auto-generated).

## 🆕 New hardware without a page
- [ ] `<source path>` (added <short-sha>) — no page found under `docs/`. Suggest: `docs/<section>/<page>.md`.

## ♻️ Stale decisions
- [ ] `<doc page>` still says "<old>"; `<source file>` changed it to "<new>" (<short-sha>).

## ⚠️ Contradictions
- [ ] `<doc page>` conflicts with `<source file>`: <one line>.

## Notes
- Build-page cross-check (GitHub Projects #2): <done / skipped>.
- Commits reviewed: <range or shas>.
```

Keep each item to one actionable line that cites the source commit/file and the target
page. Group empty sections may be omitted.

## Maintaining this playbook

Improve the review by editing **this file** in a PR — the scheduled task is a thin pointer
to it and does not need to be recreated. Local build/preview steps for verifying any fix
live in [`../../docs/_source/docs-site-workflow.md`](../../docs/_source/docs-site-workflow.md).
